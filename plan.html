0 · Phạm vi (Scope)

Mục tiêu cốt lõi: nhận text → trả về JSON lệnh hợp lệ, đóng gói thành API. Frontend (nút micro + STT) chỉ là client để test API có chạy đúng không, không phải trọng tâm. Điều khiển xe vật lý tách rời, không bắt buộc cho MVP.

Ưu tiên build & test theo thứ tự: ① API (NestJS) → test bằng curl/Postman → ② client web chỉ để demo end-to-end.

3 · Luồng logic chi tiết

Giai đoạn 1 — Giao diện & ghi âm (Next.js, chỉ để test)

  • Màn hình chính: 1 nút micro nổi bật, trạng thái mặc định “Sẵn sàng”.
  • Người dùng nhấn giữ / click để nói → hiện hiệu ứng sóng âm (visualizer) làm feedback.
  • Dùng MediaRecorder ghi audio (vd .webm/.wav) — không dùng Web Speech API nữa. STT sẽ do backend gọi Groq Whisper (xem Q2).
  • Kết thúc: thu được file audio blob để gửi lên API.

Giai đoạn 2 — Gọi API (Client → Server)

  • POST /api/v1/robot/command/audio (multipart) đính kèm file audio + header X-API-Key.
  • (Hoặc test nhanh không cần ghi âm: POST /command với { text } sẵn.)

Giai đoạn 3 — STT + Prompt engineering (NestJS)

  • Guard kiểm tra X-API-Key khớp env API_KEY → sai/thiếu trả 401.
  • STT: NestJS gọi Groq audio/transcriptions (model whisper-large-v3, language: "vi") → nhận text.
  • Bọc text vào System Prompt định nghĩa sẵn các action + tham số hợp lệ.
SYSTEM: Bạn là bộ điều khiển robot. Phân tích câu nói và trả về MẢNG JSON
các bước di chuyển. Action cho phép: forward, backward, left, right, stop.
Mỗi bước có "action" và "duration" (giây). KHÔNG giải thích thêm.

USER: cho xe chạy tới một đoạn rồi quẹo phải

Giai đoạn 4 — LLM → JSON (Groq llama-3.3-70b-versatile)

  • Gọi Groq chat completions với response_format: { type: "json_object" }, temperature: 0 để ép trả JSON thuần.
  • NestJS parse + validate bằng Zod. Nếu sai schema → retry / trả lỗi rõ ràng (xem Q4).
  • (Tùy chọn) Ghi log vào Supabase.
  • Trả JSON về frontend hiển thị.

4 · Hợp đồng dữ liệu (API & JSON)

Request A — gửi audio (luồng chính)

POST /api/v1/robot/command/audio
Content-Type: multipart/form-data
X-API-Key: <api_key>

file: <blob audio .webm/.wav>   # backend gọi Groq Whisper → text → LLM

Request B — gửi text (test nhanh)

POST /api/v1/robot/command
Content-Type: application/json
X-API-Key: <api_key>

{ "text": "cho xe chạy tới một đoạn rồi quẹo phải" }

Auth MVP: 1 Guard so sánh header X-API-Key với env API_KEY. Sai/thiếu → 401. Cả 2 endpoint dùng chung lõi text → LLM → JSON; endpoint audio chỉ thêm bước STT (Groq Whisper) ở đầu.

Response (thành công) — MVP

{
  "status": "success",
  "original_text": "cho xe chạy tới một đoạn rồi quẹo phải",
  "commands": [
    { "action": "forward", "duration": 2 },
    { "action": "right",   "duration": 1 }
  ]
}

MVP chỉ dùng tham số cơ bản (action + duration, tất cả time-based, không cần cảm biến). Bộ tham số đầy đủ là phần mở rộng sau.

Response (lỗi / không hiểu)

{
  "status": "error",
  "original_text": "...",
  "reason": "Không nhận diện được hành động hợp lệ",
  "commands": []
}

Bộ tham số đầy đủ của mỗi command → xem phần 4b · Tham số khối di chuyển ngay bên dưới.

4b · Tham số khối di chuyển (research)

Mình research đầy đủ rồi rút xuống mức cơ bản cho MVP. Hai chuẩn điều khiển differential-drive (đúng mô hình EV3 / xe Arduino) tham khảo:

  • EV3 (Move Steering): power (−100…100) + steering (−100…100) + chế độ dừng seconds / degrees / rotations. Steering ±100 = xoay tại chỗ, ±50 = vòng cung, 0 = thẳng.
  • Kinematic: vận tốc dài v + vận tốc góc ω, bán kính cung R (R=0 = xoay tại chỗ).
✅ MVP — chỉ 2 trường cơ bản Dùng cái này
Tất cả time-based → không cần encoder / cảm biến / la bàn. Hợp với giọng nói mơ hồ ("chạy một đoạn", "quẹo phải"). Rẽ cũng tính bằng giây thay vì độ.
Tham sốKiểu · đơn vịBắt buộcÝ nghĩa

durationnumber · giây✓ (trừ stop)Chạy/rẽ trong bao nhiêu giây. stop thì không cần.

Zod schema — MVP

const Command = z.object({
  action:   z.enum(['forward','backward','left','right','stop']),
  duration: z.number().positive().optional(),  // giây; bỏ trống cho stop
});
const Commands = z.array(Command);

5 · Open Questions (tất cả đã chốt ✅)

Q1 Groq Llama 3.3 70B · Q2 Groq Whisper · Q3 JSON cơ bản · Q4 retry 1 lần · Q5 API key · Q6 tiếng Việt + 5 action cố định.

Q1

Model LLM phân tích ý định? ĐÃ CHỐT

Quyết định: Groq llama-3.3-70b-versatile — chung provider/key với STT, hiểu tiếng Việt + bám JSON tốt, rất nhanh trên Groq.
  • Tham số: response_format: { type: "json_object" }, temperature: 0 để output ổn định.
  • Giá: ~$0.59/$0.79 per 1M token (in/out), context 131K — thừa cho prompt ngắn.
  • Khởi đầu rẻ hơn: có thể tạm dùng llama-3.1-8b-instant ($0.05/$0.08) rồi nâng lên 70B nếu thiếu chính xác.
  • ⚠️ Tránh qwen3-32b & llama-4-scout-17b (đã deprecate 17/06/2026).
Nên bọc trong 1 service trừu tượng để đổi model/provider sau dễ dàng.
Q2

Speech-to-Text lấy ở đâu? ĐÃ CHỐT

Quyết định: Groq Whisper (server-side). Frontend ghi audio bằng MediaRecorder → gửi lên NestJS → NestJS gọi Groq audio/transcriptions.
  • Model: whisper-large-v3 (WER 10.3%, chính xác hơn — lệnh điều khiển xe error-sensitive). Muốn rẻ/nhanh hơn thì whisper-large-v3-turbo ($0.04/giờ).
  • Tham số: language: "vi", temperature: 0, response_format: "json". Dùng endpoint transcriptions (giữ tiếng Việt), KHÔNG dùng translations (chỉ ra tiếng Anh).
  • Chi phí: có free tier (rate-limit, file ≤ 25MB) đủ cho demo; tối thiểu tính phí 10s/request.
  • API key Groq chỉ để ở backend, không lộ ra frontend.
Lý do bỏ Web Speech API: tiếng Việt không ổn định, phụ thuộc trình duyệt (Safari/Firefox kém).
Q3

JSON command chi tiết tới đâu? ĐÃ CHỐT

Quyết định: MVP chỉ action + duration (xem 4b). Mọi tham số nâng cao (speed, angle, radius, distance...) để dạng optional, mở rộng khi nối phần cứng thật.
Q4

Xử lý khi LLM trả JSON sai? ĐÃ CHỐT

Quyết định: Zod validate + retry 1 lần (gọi lại Groq kèm thông báo lỗi để self-correct). Vẫn fail → trả status: "error". Luôn trả response cấu trúc cố định để frontend xử lý đồng nhất.
Q5

Auth & Database cho MVP? ĐÃ CHỐT

Quyết định: chỉ check API key ở header X-API-Key (so với env API_KEY). Không Firebase, không Supabase ở giai đoạn này. Bật sau khi cần multi-user / lưu lịch sử / macros.
Q6

Ngôn ngữ & phạm vi câu lệnh? ĐÃ CHỐT

Quyết định: chỉ tiếng Việt + tập action cố định (closed set) gồm 5 lệnh: forward, backward, left, right, stop.
  • "Tập action cố định" = LLM chỉ được chọn trong 5 action này (ép bằng z.enum([...])), không được bịa lệnh mới → validate chắc chắn + an toàn cho xe.
  • Câu nói ngoài phạm vi (vd "hát một bài") → trả status: "error", KHÔNG sinh lệnh bậy.
  • STT set language: "vi"; System Prompt liệt kê rõ 5 action cho phép.
Đa ngôn ngữ (Việt + Anh) để mở rộng sau nếu cần.

7 · Rủi ro & lưu ý

  • Định dạng audio: MediaRecorder thường ra .webm/opus — Groq có hỗ trợ, nhưng nên test; cần thì convert sang .wav. File ≤ 25MB (free tier).
  • Chi phí Groq: tối thiểu tính phí 10s/request; free tier có rate-limit. Câu lệnh ngắn vẫn ổn cho demo.
  • Chi phí LLM: mỗi lệnh = 1 STT + 1 LLM call. Cần rate-limit / chọn model rẻ (GPT-4o-mini, Gemini Flash).
  • LLM "bịa" duration: câu nói mơ hồ ("một đoạn") → model tự đoán số giây. Cần System Prompt quy ước giá trị mặc định.
  • Bảo mật API key: key Groq & LLM CHỈ để ở backend (NestJS), không bao giờ lộ ra frontend.
  • Quyền micro: trình duyệt yêu cầu HTTPS + cấp quyền; xử lý trường hợp người dùng từ chối.
  • Độ trễ: tổng = upload audio + STT (Groq) + LLM. Groq STT rất nhanh; hiển thị loading rõ ràng.

8 · Bước kế tiếp · Lớp thực thi (đa nền tảng) — bridge đầu tiên: DJI (sau MVP)

MVP dừng ở JSON lệnh hợp lệ. Bước kế tiếp: biến JSON đó thành chuyển động thật. Vẫn giữ nguyên tắc gốc — “hiểu ngôn ngữ” tách rời “thực thi vật lý” — nhưng đẩy mạnh hơn: backend là dịch vụ trung lập, phục vụ NHIỀU nền tảng robot (DJI, EV3, Arduino…). Backend KHÔNG biết gì về DJI; nó chỉ xuất JSON lệnh trừu tượng.

Bridge là một service hoàn toàn TÁCH RỜI backend (low coupling). Mỗi nền tảng robot có một bridge riêng, tự dịch JSON chung → SDK của nó. Thêm/đổi robot = viết bridge mới, không đụng vào backend.

  • SDK (bridge DJI): thư viện Python chính thức của DJI — robomaster (RoboMaster EP / S1). Cài: pip install robomaster.
  • Kết nối Wi-Fi: hai chế độ — router mode (robot + laptop cùng router) hoặc direct mode (laptop nối thẳng access-point của robot). Bridge khởi tạo bằng robot.initialize(conn_type="sta"|"ap").
  • Cách nhận JSON: bridge mở 1 HTTP endpoint nhỏ (FastAPI) POST /execute nhận { "commands": [...] } — đúng response của API ở §4. Client (web) sau khi nhận JSON từ backend sẽ POST sang URL của bridge; backend đứng ngoài. Sau có thể đổi sang MQTT/WebSocket nếu cần.
  • Vocabulary chung: JSON từ backend là tập lệnh trừu tượng (vd 5 action MVP, sau thêm fire/aim…). Bridge nào không hỗ trợ 1 action thì bỏ qua + log, không lỗi cả chuỗi.

Ánh xạ lệnh: JSON action → DJI SDK (time-based)

API xuất mỗi lệnh kèm speed (0–100%) và một trong seconds/degrees/rotations. Bridge time-based nên dùng chassis.drive_speed() (scale theo speed%) + chờ theo thời gian + dừng — KHÔNG dùng chassis.move(). degrees/rotations (độ quay bánh) được xấp xỉ ra thời gian vì bridge không có encoder.

actionBộ phậnGọi SDK (ý niệm)
forwardchassis (khung gầm)drive_speed(x=+SPEED·f, y=0, z=0) → chờ theo thời lượng
backwardchassisdrive_speed(x=-SPEED, y=0, z=0)
leftchassisdrive_speed(x=0, y=0, z=-TURN)  (z = tốc độ quay °/s)
rightchassisdrive_speed(x=0, y=0, z=+TURN)
stopchassisdrive_speed(0,0,0) / chassis.stop()

SPEED (vd 0.5 m/s) & TURN (vd 90 °/s) là tốc độ ở 100%, scale theo speed% (f). Dấu của z (trái/phải) tùy chiều trục thực tế — kiểm tra khi test, đừng tin lý thuyết.

Khung code bridge (phác thảo)

from robomaster import robot
import time

ep = robot.Robot()
ep.initialize(conn_type="sta")          # cùng Wi-Fi với robot
chassis = ep.chassis
SPEED, TURN = 0.5, 90                    # m/s , °/s  (ở 100% speed)

def run(commands):
    for c in commands:
        a = c["action"]
        f = (c.get("speed", 60)) / 100   # speed% → hệ số 0..1
        if   a == "forward":  chassis.drive_speed(x= SPEED*f, y=0, z=0)
        elif a == "backward": chassis.drive_speed(x=-SPEED*f, y=0, z=0)
        elif a == "left":     chassis.drive_speed(x=0, y=0, z=-TURN*f)
        elif a == "right":    chassis.drive_speed(x=0, y=0, z= TURN*f)
        elif a == "stop":     chassis.drive_speed(0, 0, 0); continue
        time.sleep(seconds_of(c))        # seconds | degrees/rotations→xấp xỉ | mặc định 1
        chassis.drive_speed(0, 0, 0)     # dừng giữa mỗi bước cho an toàn
    chassis.drive_speed(0, 0, 0)         # chốt: luôn dừng khi xong

Bọc run() sau endpoint FastAPI POST /execute: trả 202 ngay (fire-and-forget) rồi chạy run() ở background task. Validate lại bằng cùng schema (action hợp lệ, thời lượng ≤ trần) trước khi truyền cho robot — không tin input mù quáng.

Rủi ro & lưu ý lớp thực thi

  • ⚠️ SDK với RoboMaster S1: lib robomaster chính thức hỗ trợ EP; S1 thường phải bật "SDK mode" (qua app RoboMaster / mở khóa chế độ kết nối) thì initialize() mới nối được. Xác minh kết nối ở E1 trước; trong lúc đó dùng BRIDGE_DRY_RUN=1 để dev mọi thứ khác.
  • An toàn vật lý #1: luôn có E-STOP + auto-stop sau mỗi bước; giới hạn duration tối đa (vd ≤ 10s) ngay trong bridge. Robot chạy thật → lỗi phần mềm = hư hỏng/va chạm thật.
  • Mạng Wi-Fi: robot & PC phải cùng mạng. Direct mode ổn định hơn nhưng PC mất internet. Mất kết nối giữa chừng → robot phải tự dừng (watchdog).
  • Độ trễ cộng dồn: STT + LLM + mạng → lệnh tới robot trễ vài giây. Hợp lệnh tuần tự (“tiến rồi quẹo phải”), KHÔNG hợp điều khiển realtime kiểu joystick.
  • Mở rộng Súng / Gimbal / Camera: thêm action mới (vd fire, aim, look) → map sang ep.blaster.fire(), ep.gimbal.moveto(), ep.camera. Nhớ mở rộng cả z.enum([...]) ở API (§4b) cho khớp.
